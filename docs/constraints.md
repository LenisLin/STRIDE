# Constraints (Locked)

These constraints govern STRIDE as the live project/scientific surface. The
canonical core-package direction is `src/stride/`.

C1. Repo-as-memory: locked constraints live here; undocumented constraints are not binding.
C2. Strict Configuration Decoupling: The reusable library layer (`src/stride/`) is STRICTLY FORBIDDEN from importing yaml or reading config files directly. The repository MAY provide config template files, but they exist only as task-layer instantiation references. All concrete path decisions, config reads, config interpretation, hyperparameters, and runtime configurations must be performed in `tasks/task_*/` (or equivalent task-scoped entrypoints) and then passed into the library layer as explicit function arguments.
C3. No silent failures: logging + validation + fail-fast assertions are mandatory. Programmer-level contract violations MUST fail fast via canonical `ContractError` in low-level engines. Canonical trainer runtime MUST fail fast on non-finite objective evaluations; gradient-level runtime guards may vary by implementation and are not themselves a frozen scientific contract requirement. Finite optimizer exits that exhaust the canonical main-stage step budget remain successful realized fits and MUST carry an explicit optimizer-exit flag rather than being silently downgraded to optimizer-level deferred status. Per-item batched data degeneracies (empty mass/support, numerical instability) MUST be isolated with explicit fit-status diagnostics and NaN metrics, then bypassed with structured audit logs at the pipeline level. Numerical padding (for example adding epsilon to empty vectors to force convergence) is strictly forbidden.
C4. No speculative science: uncertain algorithms/metrics must go through Tier-2 gate.
C5. Python→R bridge: any handover file must be created via `write_r_handover()`. Output directory, file name, primary key, and export format are task-scoped decisions and must be passed explicitly; the reusable library layer does not discover them from yaml/config.
C6. Domain-Logic Isolation (Library vs. Tasks): The physical boundary is two-tier on the live side: reusable core library (`src/stride/`) and task layer (`tasks/`). The reusable core MUST remain task-insensitive and may contain both method kernel modules and method-defining core inference modules of STRIDE itself; `tasks/` MUST own cohort-specific modeling choices, study design, inclusion/exclusion rules, config parsing, explicit cohort/patient/visit looping, endpoint/covariate/subgroup orchestration, reporting, and benchmark-specific evaluation. Hard-coded cohort names, benchmark-specific clinical semantics, task-specific formula construction, and reporting logic are forbidden in the reusable core. Benchmark code belongs under `tasks/` or `tasks/_shared/benchmarks/`.
C7. Representation Contract: The reusable core may contain official representation utilities and high-level validation for the official AnnData route. Raw single-cell or spot modalities MUST NOT enter observation-discrepancy operators directly. Any official or custom representation route MUST terminate in canonical STRIDE-ready objects on a shared `K`-state basis, even if the implementation route uses prototypes internally. In the current first-pass official route, that shared basis MUST be built tissue-agnostically before any tissue/domain stratification: per-cell subtype labels -> within-ROI kNN neighborhood subtype proportion vectors -> k-means shared community states, with user-configurable `k` and documented first-pass default `20`. Tissue/domain labels MUST NOT be injected into community-state construction. The current first-pass ROI/FOV observation input is then a normalized community-composition vector over that basis together with separate uniform observation mass (`mass = 1`, `mass_mode = "uniform"`). A cost matrix of shape `[K, K]` on that same basis, required global scaling objects such as `s_C` where mandated by the method, and a compatible route to valid observation-layer calibration controls of shape `[N]` when task-specific calibration is required remain required. In the official route, local representation construction, shared-basis construction, and official cost geometry construction are coupled library-supported steps that MUST remain geometrically consistent; users following the official route do not manually invent `cost_matrix`. Density mode depends on ROI area metadata and MUST preserve declared physical cells/mm² semantics where relevant for future/custom routes. Custom routes may replace raw-data-to-state-space mapping, aggregation, or cost construction, but may not redefine the mathematical meaning of canonical STRIDE inputs.
C8. Task-Driven Calibration Order: The task layer defines baseline subsets, grouping context, pairing metadata, and any observation-domain stratification first. Library calibration then uses that task-compatible context to derive the required observation-layer calibration controls; after that, the task layer assembles final solver batches `[N, K]`, and the library runs observation-layer comparison on those ROI-/sample-level tensors.
C9. Baseline Comparator Requirement: Any task claiming added value from STRIDE MUST define explicit baseline comparators. At minimum, the baseline analysis MUST include a simple state-abundance comparator and MUST distinguish what the baseline already answers from what STRIDE additionally explains.
C10. Same-Scale and Task-Scoped Baseline Rule: Baseline state-abundance comparators MUST be computed on the same physical or pseudo-mass scale as the canonical mass table used by STRIDE in that task. If STRIDE uses density, the baseline MUST use that same density scale; presenting density-based STRIDE results against counts- or proportion-based baselines as if they were method-equivalent is forbidden. It is also forbidden to present simple abundance changes as if they were uniquely discoverable only by STRIDE. Baseline evaluation, reporting, and any downstream baseline tables remain task-scoped rather than core-library-level.
C11. Patient-Relation Semantics: The canonical patient object is `(T_p, e_p)` with `T_p = [A_p | d_p]`. Docs and exported semantics MUST treat each source row `[A_{p,i,*}, d_{p,i}]` as source-side accounting on a simplex satisfying `sum_j A_{p,ij} + d_{p,i} = 1`. `d_p` is source-side outgoing open tendency. `e_p` is bounded target-side incoming open-entry tendency with `0 <= e_{p,j} <= 1`. The v1 composition-scale target-side reconstruction uses `raw_post_p = q_p^- @ A_p + e_p` and `predicted_q_p^+ = normalize(raw_post_p)`. If a normalized conditional kernel is discussed, it MUST be documented as a derived auxiliary object from `A_p` and `d_p`, not as the canonical patient object itself.
C12. Burden/Composition Separation: `mu_p^-`, `mu_p^+`, `m_p^(d)`, and `m_p^(e)` live on a patient-level pseudo-mass / burden scale on the shared `K`-state basis. `q_p^-` and `q_p^+` are derived normalized compositions only. Conservation language is limited to a soft burden-consistency anchor; docs and reports MUST weaken or disable burden-level claims when coverage or platform comparability is weak.
C13. Observation-Layer Canonical Object: The canonical observation layer in the current first pass is a domain-stratified bag-of-FOV empirical measure over ROI/FOV community-composition vectors with equal ROI/FOV mass (`mass = 1`, `mass_mode = "uniform"`). Canonical v1 `L_obs` uses the fixed, versioned, auditable `D_obs^BalancedSinkhornDivergence-v1` operator to compare the model-implied predicted target-side FOV bag with the observed target-side FOV bag. `D_obs` is a fixed operator inside `L_obs`, not a sixth loss term and not an independently weighted component. For each source FOV, `y_hat_f = normalize(v_source_f @ A_p + e_p)`. `C_norm = C_raw / s_C` remains the state/community-level cost. The FOV-level ground cost is `G[f,g] = debiased balanced Sinkhorn composition distance(y_hat_f, y_true_g; C_norm)`, and the outer loss is the debiased balanced Sinkhorn divergence under `G_norm = G / s_G_init`. `s_G_init` is fixed per evidence block from positive finite initialization-time FOV-level costs at deterministic identity-plus-small-open initialization, uses `1.0` with recorded floor usage if no positive finite costs exist, and MUST NOT be recomputed dynamically during optimization. The canonical operator is torch-native, `float64`, log-domain, differentiable, balanced, and debiased, with fixed v1 inner and outer epsilon schedules `(0.5, 0.2, 0.1)`, `max_iter = 100` per epsilon stage, `tol = 1e-6`, and `warning_tol = 1e-4`. Its canonical runtime is a fixed-iteration active-mask execution on device that freezes items once they first satisfy `residual <= tol` and records scalar convergence diagnostics only after the epsilon stage completes. Tiny negative inner composition-distance values in `[-1e-10, 0)` are clamped to `0` with provenance; values below `-1e-10` are numerical failure. Empty source or target FOV bags, invalid simplexes, negative entries, NaN/Inf values, invalid cost matrices, or unavailable `torch` for the canonical full estimator MUST fail explicitly. Numerical padding remains forbidden. Reaching `max_iter` with finite values is usable with warning; updates larger than `warning_tol` are recorded convergence warnings on diagnostic/profiling surfaces, while production optimizer forwards may materialize only compact objective/provenance fields. Canonical `L_obs` has no observation-layer unbalanced unmatched residual; open behavior is expressed only by fitted biological `d/e`. Legacy UOT, `D_pos/B_pos`, and unbalanced observation residual diagnostics may appear only as diagnostic or comparator surfaces, including the Task A Block 3 `uot_baseline` external comparator name. Docs and reports MUST keep ROI/FOV community composition distinct from the cell-level neighborhood subtype composition used during state construction, and they MUST NOT present multinomial, Dirichlet-multinomial, logistic-normal, or dense transport-plan framing as the canonical biological ontology.
C14. No State-Domain Double Counting: Domain/compartment labels are observation-layer stratification variables, grouped discrepancy variables, patient-relation input grouping variables, covariates, or analysis surfaces only. The shared `K`-state basis MUST NOT encode domain identity and then condition on domain again. Domain labels MUST NOT define state geometry or the axes of `A_p`, `d_p`, and `e_p`. Task-local ordered tissue-domain analyses, including Task A, do not redefine canonical state identity.
C15. Current Uncertainty Surface: The current core-package uncertainty meaning is bootstrap/sampling-variance uncertainty over fitted patient relation outputs (`A_p`, `d_p`, `e_p`). The current first-pass core contract uses bootstrap/sampling-variance uncertainty terminology.
C16. Public Full-Estimator Surface: `fit_stride(...)` is the live public full-estimator method surface. Its target contract is the objective-driven fit of patient-level `A_p`, `d_p`, and `e_p` with compact provenance under the frozen full-estimator objective. The public beta surface keeps explicit source/target/K semantics but does not expose reference optimizer protocol knobs such as `lr`, `max_steps`, or `min_steps`.
C17. Source/Target Declaration And Canonical Estimator Invariance: The task layer owns source/target/domain declaration, config parsing, and comparison-plan instantiation. The reusable core receives resolved source/target observation evidence blocks, does not read YAML/config directly, and does not infer task-specific biological relation semantics. Once explicit source/target inputs are passed, `fit_stride(...)` applies the same task-insensitive full STRIDE estimator and canonical `D_obs^BalancedSinkhornDivergence-v1` observation discrepancy operator. Domain is recorded in the resolved comparison-plan provenance but does not enter the core as a loss axis, state axis, relation axis, or recurrence axis. Observation-layer diagnostics may be emitted as observation evidence, but reports MUST distinguish them from biological `d/e`. Biological process claims require cohort consensus recurrence/support/dispersion over fitted patient-level `A/d/e`.
C18. Geometry/Locality Cost Contract: Official and custom representation routes that enter canonical observation comparison or full-estimator geometry/locality fitting MUST provide a finite, nonnegative, symmetric `[K, K]` shared-state cost matrix with diagonal entries equal to `0` within declared numerical tolerance. `C_norm = C_raw / s_C`, where the default `s_C` is the median of positive finite off-diagonal entries of `C_raw`. A valid positive off-diagonal scale MUST exist; otherwise the route fails the contract explicitly. Invalid state geometry MUST NOT fall back to `s_C = 1.0` and MUST NOT silently disable geometry. `L_geometry` acts on raw canonical `A_p` as `L_geometry_raw(p) = (1 / K) * sum_i sum_j A_p[i,j] * C_norm[i,j]`, with all `K` source rows included and diagonal self-retention carrying zero geometry cost. The full-estimator ledger records `L_geometry_raw`, `scale_geometry`, `normalized_L_geometry`, and `L_geometry_effective`. The optimizer geometry term is `L_geometry_effective = geometry_effective_weight * normalized_L_geometry` under the reference three-block objective.
