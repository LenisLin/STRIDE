# Constraints (Locked)

C1. Repo-as-memory: locked constraints live here; undocumented constraints are not binding.
C2. Strict Configuration Decoupling: The library layer (src/slotar/) is STRICTLY FORBIDDEN from importing yaml or reading config files directly. The repository MAY provide config template files, but they exist only as task-layer instantiation references. All concrete path decisions, config reads, config interpretation, hyperparameters, and runtime configurations must be performed in `tasks/task_*/` (or equivalent task-scoped entrypoints) and then passed into `src/slotar/` as explicit function arguments.
C3. No silent failures: logging + validation + fail-fast assertions are mandatory. Programmer-level contract violations MUST fail fast via `DataContractError` in low-level engines; per-item batched data degeneracies (empty mass/support, numerical instability) MUST be isolated with explicit `uot_status` and NaN metrics, then bypassed with structured audit logs at the pipeline level. Numerical padding (e.g., adding epsilon to empty vectors to force convergence) is strictly forbidden.
C4. No speculative science: uncertain algorithms/metrics must go through Tier-2 gate.
C5. Python→R bridge: any handover file must be created via `save_for_r()` in `src/slotar/io/bridge.py`. Output directory, file name, primary key, and export format are task-scoped decisions and must be passed explicitly; `src/slotar/` does not discover them from yaml/config.
C6. Domain-Logic Isolation (Library vs. Tasks): The physical boundary is strictly two-tier: `src/slotar` and `tasks/`. `src/slotar` MUST remain benchmark-agnostic and may contain both method kernel modules and method-defining core inference modules of SLOTAR itself; `tasks/` MUST own cohort-specific modeling choices, study design, inclusion/exclusion rules, config parsing, explicit cohort/patient/visit looping, endpoint/covariate/subgroup orchestration, reporting, and benchmark-specific evaluation. Hard-coded cohort names, benchmark-specific clinical semantics, task-specific formula construction, and reporting logic are forbidden in `src/slotar`.
C7. Representation Contract: `src/slotar` may contain official representation utilities and high-level validation for the official AnnData route. Raw single-cell or spot modalities MUST NOT enter UOT directly. Any official or custom representation route MUST terminate in canonical SLOTAR-ready objects on a shared state/prototype axis: ROI-/sample-level nonnegative mass distributions over that axis, explicit `mass_mode` semantics (`count`, `density`, or `proportion`) with no silent mixing, a cost matrix of shape `[K, K]` on that same axis, required global scaling objects such as `s_C` where mandated by the method, and a compatible route to valid `lambda_pl` of shape `[N]`. In the official route, local representation construction, shared state/prototype construction, and official cost geometry construction are coupled library-supported steps that MUST remain geometrically consistent; users following the official route do not manually invent `cost_matrix`. Density mode depends on ROI area metadata and MUST preserve declared physical cells/mm² semantics where relevant. Custom routes may replace raw-data-to-state-space mapping, aggregation, or cost construction, but may not redefine the mathematical meaning of the canonical SLOTAR inputs.
C8. Task-Driven Calibration Order: The task layer defines baseline subsets, grouping context, and pairing metadata first. Library calibration then uses that task-compatible context to derive lambdas or a valid path to `lambda_pl`; after that, the task layer assembles final solver batches `[N, K]`, and the library runs batched UOT on those ROI-/sample-level tensors.
C9. Baseline Comparator Requirement: Any task claiming added value from SLOTAR MUST define explicit baseline comparators. At minimum, the baseline analysis MUST include a simple state-abundance comparator and MUST distinguish what the baseline already answers from what SLOTAR additionally explains.
C10. Same-Scale and Task-Scoped Baseline Rule: Baseline state-abundance comparators MUST be computed on the same physical mass scale as the canonical mass table used by SLOTAR in that task. If SLOTAR uses density, the baseline MUST use that same density scale; presenting density-based SLOTAR results against counts- or proportion-based baselines as if they were method-equivalent is forbidden. It is also forbidden to present simple abundance changes as if they were uniquely discoverable only by SLOTAR. Baseline evaluation, reporting, and any downstream baseline tables remain task-scoped rather than core-library-level.
